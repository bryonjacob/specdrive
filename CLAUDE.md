# specdrive

Language-agnostic CLI for Gherkin behavioral contract management. Manages shared spec packages, audits RID integrity, and verifies bidirectional traceability between specs and test results.

## Purpose

Any project that uses Gherkin specs with `@RID-*` tags can use specdrive — regardless of programming language, test framework, or whether they use property-based testing.

## Architecture Overview

CLI tool with five commands (`init`, `install`, `audit`, `verify`, `status`). No framework — direct `process.argv` dispatch. Two-layer design: pure logic modules (tested at 96%+ coverage) and orchestration modules (integration-level, use child_process for git).

## Module Index

### Core

- `src/types.ts` - All interfaces (SpecdriveConfig, SourceUri, SpecManifest, Lockfile, RidEntry, AuditResult)
- `src/sources.ts` - URI parsing (path:, github:, gist:, git:) and clone URL generation
- `src/config.ts` - Load/validate specdrive.yaml (`specs` and/or `localSpecs`)
- `src/scan.ts` - Walk .feature files, extract @RID-\* tags with line numbers and a per-file `scenarioId` (incremented at each `Scenario:` / `Scenario Outline:` boundary, used by audit to distinguish same-scenario duplicates from cross-scenario reuse); `scanLocalSpecs()` for local spec dirs. Tags are harvested only from genuine Gherkin tag lines (trimmed line of whitespace-separated `@`-tokens); RIDs in `#` comments or inside `"""` / ``` docstrings are intentionally ignored.
- `src/lockfile.ts` - Read/write .specdrive-lock.yaml
- `src/resolve.ts` - Load spec.yaml manifests, apply extends/omits
- `src/color.ts` - ANSI color utilities (fmt, pad, box), respects NO_COLOR and TTY detection
- `src/rid.ts` - Centralized RID regex patterns (RID_TAG_RE, RID_RE, RID_FORMAT_RE)
- `src/fs-utils.ts` - Shared filesystem utilities (fileExists, fileContains, walkFeatureFiles)
- `src/audit.ts` - Audit logic: validate installed + local specs, RID format, duplicates. Duplicate detection is scenario-aware: the requirement↔scenario relationship is many-to-many, so the same RID on distinct scenarios (keyed by `spec/file/scenarioId/rid`) is valid; only a RID repeated within one scenario is an error.

### Commands

- `src/index.ts` - CLI entry point (shebang, argv dispatch)
- `src/fetch.ts` - Git clone / local copy via child_process (orchestration)
- `src/install.ts` - `install` command orchestration
- `src/verify.ts` - `verify` command: cross-reference vendored + local RIDs against JUnit XML. Colorized output with source locations.
- `src/status.ts` - `status` command: print spec table (vendored + local)

### Init System (framework adapter architecture)

- `src/init/types.ts` - `Detection`, `AdapterFile`, `ConfigEdit`, `Adapter` interfaces
- `src/init/detect.ts` - Runs all adapter detectors, ranks by confidence
- `src/init/index.ts` - `specdrive init` orchestrator: detect → plan → show → apply → summarize
- `src/init/adapters/index.ts` - Adapter registry (add a framework = add here)
- `src/init/adapters/pytest.ts` - Emits conftest.py with `pytest_collection_modifyitems` hook
- `src/init/adapters/vitest.ts` - Emits specdrive.setup.ts + wires into vitest.config.ts
- `src/init/adapters/generic.ts` - Fallback: prints JUnit XML format spec for unknown frameworks

### Adding a New Framework Adapter

1. Create `src/init/adapters/<framework>.ts` implementing the `Adapter` interface
2. Add it to the `adapters` array in `src/init/adapters/index.ts`
3. The orchestrator (`src/init/index.ts`) does not need to change

Each adapter provides:

- `detect(dir)` — inspect project files, return confidence level + evidence
- `files(dir)` — self-contained snippet files to create (no runtime dependency on specdrive)
- `configEdits(dir)` — modifications to existing config files (safe=true if automatable, safe=false falls back to manual instructions)

Key properties: idempotent (marker comments prevent duplication), safe (only automates edits it can parse reliably), communicative (shows plan before applying).

## File Layout (consumer-side)

```
specdrive.yaml            # config: specs (vendored) and/or localSpecs (project-local)
.specdrive-lock.yaml      # locked versions with SHAs (committed)
.specdrive/cache/         # git clones (gitignored)
specs/_specdrive/         # vendored feature files (gitignored)
conftest.py               # (pytest) or specdrive.setup.ts (vitest) — generated by `specdrive init`
```

## Config Format

```yaml
# Local spec directories (optional, relative to project root)
localSpecs:
  - specs/auth

# Vendored spec packages (optional)
specs:
  - name: core-spec
    source: github:org/spec@v1.0

# Fail on tests without RID annotations (default: false)
strictTraceability: true
```

At least one of `specs` or `localSpecs` must be present.

## Source URI Schemes

- `path:../local-spec` — local filesystem
- `github:org/repo@ref` — GitHub repo
- `gist:id@ref` — GitHub gist
- `git:url@ref` — any git URL

## Code Style

No semicolons, single quotes, trailing commas (es5), 100 char print width. Enforced by prettier + eslint with complexity <= 10.
